Step 1 of 4
1
Install Snort and Verify the Version

Snort was installed on the Ubuntu VirtualBox VM using the system package manager. After installation, the snort --version command was run to confirm the correct version was active and all required libraries were linked.

# Install Snort via apt
sudo apt-get install snort -y

# Confirm the installed version
snort --version

The output confirmed Snort 2.9.15.1 GRE (Build 15125) with libpcap 1.10.1, PCRE 8.39, and ZLIB 1.2.11 — all dependencies required for full packet inspection capability.

christabel@christabel-VirtualBox: /etc/snort/rules — Terminal
Figure 1 — Snort 2.9.15.1 GRE successfully installed. The snort --version command confirms the build number and all linked libraries (libpcap, PCRE, ZLIB).

Why This Matters

Version verification is a fundamental step in any tool deployment. In a real SOC, running the wrong version of an IDS can mean missing detection signatures, having incompatible rule formats, or leaving known vulnerabilities in the sensor itself unpatched.

Step 2 of 4
2
Configure snort.conf — Network Variables

The Snort configuration file at /etc/snort/snort.conf was opened using sudo nano snort.conf. The key network variables were reviewed and set to match the home lab network topology.

# Open the Snort configuration file
sudo nano /etc/snort/snort.conf

# Key variables configured inside the file:
ipvar HOME_NET any
ipvar EXTERNAL_NET any
ipvar DNS_SERVERS $HOME_NET
ipvar HTTP_SERVERS $HOME_NET
ipvar SQL_SERVERS $HOME_NET

HOME_NET defines the network Snort is defending. Setting it to any in the lab means Snort will monitor all traffic regardless of direction — appropriate for a controlled lab environment where all traffic is trusted test traffic.

GNU nano 6.2 — snort.conf
Figure 2 — The snort.conf file open in nano. Network variables HOME_NET and EXTERNAL_NET are visible, along with server group definitions for DNS, HTTP, SQL, SMTP, Telnet, SSH, and FTP services.

Why This Matters

Correct HOME_NET configuration is critical. If this is wrong, Snort will either miss inbound attacks (because it does not know what to protect) or generate floods of false positives from internal traffic. In a production SOC, this value is set to the specific subnets of the organisation.

Step 3 of 4
3
Write Custom Detection Rules in local.rules

Five custom detection rules were written in /etc/snort/rules/local.rules. These rules target ICMP ping activity and four common Nmap scan techniques used by attackers during the reconnaissance phase of an attack.

# ICMP Ping detection
alert icmp any any -> any any (msg:"ICMP Ping Detected"; itype:8; sid:1000001; rev:1;)

# Nmap TCP SYN scan (half-open)
alert tcp any any -> $HOME_NET any (msg:"NMAP SYN SCAN detected"; flags:S; threshold:type both, track by_src, count 10, seconds 1; sid:1000002; rev:1;)

# Nmap TCP Connect scan
alert tcp any any -> $HOME_NET any (msg:"NMAP TCP Scan Detected"; flags:S; threshold:type both, track by_src, count 10, seconds 1; sid:1000002; rev:1;)

# Nmap NULL scan (no flags set)
alert tcp any any -> $HOME_NET any (msg:"NMAP NULL Scan Detected"; flags:0; sid:1000003; rev:1;)

# Nmap FIN scan
alert tcp any any -> $HOME_NET any (msg:"NMAP FIN Scan Detected"; flags:F; sid:1000004; rev:1;)

# Nmap XMAS scan (FIN+PSH+URG flags)
alert tcp any any -> $HOME_NET any (msg:"NMAP XMAS Scan Detected"; flags:FPU; sid:1000005; rev:1;)

Figure 3 — The local.rules file showing all five custom detection rules targeting ICMP pings and Nmap SYN, NULL, FIN, and XMAS scans. Each rule has a unique SID (Snort ID) starting at 1000001.

Why This Matters

Attackers routinely use Nmap during the reconnaissance phase to identify open ports, running services, and operating systems before launching an actual attack. These rules map directly to MITRE ATT&CK T1046 — Network Service Discovery. Writing rules like these teaches you how to translate threat intelligence into actionable detection logic — a core skill for any detection engineer or SOC analyst.

Step 4 of 4
4
Run Attack Simulation and Validate Alerts

With Snort running in alert mode on the Ubuntu VM, a Windows host on the same network was used to simulate attacks using Nmap 7.99 and the ping command via Windows PowerShell. The Ubuntu VM's IP address was 192.168.0.81.

# Commands run from Windows PowerShell against the Ubuntu VM (192.168.0.81)

# TCP Connect scan
nmap -Pn -sT 192.168.0.81

# TCP SYN (stealth) scan
nmap -Pn -sS 192.168.0.81

# ICMP ping sweep
ping 192.168.0.81

The dual-screen capture below shows the Windows PowerShell window running the scans on the right, with the Snort alert log on the Ubuntu VM generating real-time alerts on the left. Snort successfully detected and logged the ICMP Ping and Nmap scan activity.

Why This Matters

Alert validation closes the loop on the whole project. Writing a rule without proving it fires is not enough — in a real SOC, unvalidated rules are a liability. They may never fire at all, or they may fire on the wrong traffic. This step proves the end-to-end chain: attack occurs → packet captured → rule matched → alert generated. That chain is the foundation of any detection engineering workflow.

Reflection

What I Learned

This project gave me practical experience with one of the most widely used open-source IDS tools in the security industry. The key insight was understanding the difference between writing a rule and validating a rule. A detection that has never been tested against real traffic is not a detection — it is just a hope.

I also gained a clearer understanding of how Nmap scan types differ at the TCP flag level. A SYN scan sends only the SYN flag and never completes the handshake, which means it is harder for basic firewalls to log. A NULL scan sends no flags at all, exploiting how some older systems respond to malformed packets. A XMAS scan lights up FIN, PSH, and URG simultaneously — like plugging in a Christmas tree, hence the name. Each of these has a different detection signature, which is why separate rules are needed for each.

In a SOC environment, an analyst would see these alerts in a SIEM such as Splunk or Microsoft Sentinel. Understanding what Snort detects and why it fires — not just that it fires — is what separates a reactive alert-acknowledger from a skilled analyst who can tune rules, reduce false positives, and improve detection coverage over time.
